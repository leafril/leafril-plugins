> [!note] Summary
> **고민**
> 기존 API 응답이 데이터를 직접 반환하는 구조여서, 에러 발생 시 클라이언트가 에러 코드를 일관되게 처리할 수 없었다.
>
> **검토**
> - 기존 방식 유지 + HTTP 상태 코드만으로 에러 구분
> - envelope 패턴으로 응답을 감싸서 에러 코드를 표준화
>
> **선택**
> envelope 방식 (`{ status, data, error }`) 도입 — 에러 코드를 응답 본문에 포함시켜 클라이언트가 HTTP 상태 코드와 무관하게 일관된 에러 처리를 할 수 있도록

기존에는 API가 데이터를 바로 반환했다. 정상 응답일 때는 문제가 없지만, 에러 상황에서 클라이언트가 에러 종류를 구분하려면 HTTP 상태 코드에만 의존해야 했다. 비즈니스 로직 에러가 다양해지면서 상태 코드만으로는 세분화된 에러 처리가 어려워졌다.

envelope 구조로 전환하면 모든 응답이 동일한 형태를 가진다.

```json
{
  "status": "success",
  "data": { ... },
  "error": null
}
```

에러 응답도 같은 구조를 따른다.

```json
{
  "status": "error",
  "data": null,
  "error": {
    "code": "INVALID_INPUT",
    "message": "..."
  }
}
```

이렇게 하면 클라이언트 측에서 응답 파싱 로직을 하나로 통일할 수 있고, 에러 코드를 표준화하여 프론트엔드와 에러 처리 계약을 명확히 할 수 있다.
