# Project

기능 개발의 컨텍스트 문서를 생성하고 관리하는 스킬.

핵심 가치: **Claude Code가 "작업 이어하자"를 들었을 때, 이 문서만 읽으면 바로 시작할 수 있게 하는 것.**

이 문서는 세 가지 질문에 답한다:
1. **뭘 만들지** — 요구 사항, 정책
2. **뭐가 남았는지** — 작업 목록 (체크리스트)
3. **어떻게 해왔는지** — 관련 문서 (daybook 링크)

## 설정

`~/.claude/note-plugins.json`에서 `noteRoot`를 읽는다. 파일이 없거나 `noteRoot`가 없으면 사용자에게 물어보고 저장한다.

## 작성 규칙

파일 경로, 템플릿, frontmatter, 용어 규칙은 [references/rules.md](references/rules.md)에 정의되어 있다.
실행 시 이 파일을 읽고 그에 맞춰 작성한다.

## 워크플로우

### Phase 1: 대상 파일 결정

1. 파일 경로: `{noteRoot}/projects/{조직}/{기능명}.md`
2. 인자로 경로가 전달되면 해당 파일을 찾거나 생성
3. 인자 없이 호출되면 `projects/` 하위 파일 목록을 보여주고 선택하게 한다
4. 조직명이 불확실하면 사용자에게 물어본다
4. 기존 파일이 있으면 읽고 내용 파악, 없으면 새로 생성

### Phase 2: 작성·수정

사용자가 전달한 내용에 따라 문서를 생성하거나 수정한다.

**새 문서 생성 시:**
- references/rules.md의 템플릿에 따라 작성
- 사용자가 전달한 내용에서 요구 사항, 정책, 작업 항목을 추출

**raw 기획 문서가 입력된 경우:**

기획 문서에는 비즈니스 배경, 가설, 경쟁사 분석, UI/UX 상세 스펙, 오디오 에셋 등
개발에 직접 필요하지 않은 내용이 많다. 개발 범위에 맞는 내용만 추출한다.

- 배경/목적 → 배경 섹션으로 압축 (1-3문장)
- 요구사항/정책 → 도메인 용어로 재구성
- 가설, 경쟁사 분석, 참고 자료 → 버린다
- UI/UX 상세 스펙(화면 구성, 에셋, 애니메이션) → 프론트엔드 작업이면 남기고, 백엔드 작업이면 버린다. 판단이 어려우면 사용자에게 물어본다
- 요구사항에서 작업 목록 도출 → 체크리스트로 분해

**아이디어 수준의 입력인 경우:**

개인 프로젝트 등에서 기획 문서 없이 아이디어만 전달될 수 있다. 있는 만큼만 채우고 나머지는 비워둔다. 개발하면서 요구 사항·정책이 구체화되면 그때 업데이트한다.

**기존 문서 수정 시:**
- 작업 목록 체크/추가, 요구 사항 변경, 관련 문서 링크 추가 등
- 사용자가 말한 내용만 반영. 임의로 항목을 추가하거나 변경하지 않는다

## 판단 기준

### 파일명

기능 이름은 구체적으로. "뭘 개발하는지" 한눈에 알 수 있어야 한다.
- 좋은 예: `사용자-인증-리팩토링.md`, `알림톡-발송-조건-이원화.md`
- 나쁜 예: `backend-작업.md`, `2월-태스크.md`

### 작업 목록

체크리스트 항목은 "완료 여부를 판단할 수 있는 단위"로 작성한다.
- 좋은 예: `- [ ] 렐름 목록 조회 API`, `- [ ] 기존 데이터 마이그레이션`
- 나쁜 예: `- [ ] 백엔드 작업`, `- [ ] 코드 수정`

### 관련 문서 링크 추가

daybook 섹션 파일을 작성한 후, 해당 프로젝트 문서의 관련 문서 섹션에 링크를 추가한다.
- 중복 링크는 추가하지 않는다
- 링크 형식은 rules.md의 패턴을 따른다

## 대화 방식

- 메타데이터를 묻는 폼식 질문은 하지 않는다
- 사용자가 전달한 내용을 그대로 반영한다. 임의로 작업을 추가하지 않는다
- 요구 사항이 불충분하면 구체적으로 물어본다 ("어떤 데이터를 반환해야 하나요?" 같은)
- 파일명, frontmatter는 스스로 판단하되, 조직명이 불확실하면 물어본다
