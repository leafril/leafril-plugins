# Project 작성 규칙

## 파일 경로

- 패턴: `projects/{조직}/{기능명}.md`
- 예시: `projects/acme/사용자-인증-리팩토링.md`
- 파일명은 어떤 기능을 개발하는지 구체적으로 작성
- 프로젝트(리포) 구분은 frontmatter `project` 필드로

## 문서 템플릿

```markdown
---
project: repo-name
status: planning
---

## 배경

왜 이 기능이 필요한지. (선택 — 요구 사항만으로 맥락이 충분하면 생략)

---

## 요구 사항

추가/변경할 것. 도메인 용어로 작성.

## 정책

비즈니스 규칙, 조건, 제한. (해당할 때만)
- 예: "하루 최대 3회, 중복 지급 불가"

---

## 작업 목록

- [ ] 작업 항목 1
- [ ] 작업 항목 2
- [x] 완료된 작업

---

## 관련 문서

- [[YYYY/MM/DD/주제]]
```

## frontmatter 필드

**필수:**
- `project`: 프로젝트(리포)명
- `status`: planning / in-progress / done

**선택:**
- `start_date`: 시작 날짜 (YYYY-MM-DD)
- `end_date`: 종료 날짜 (YYYY-MM-DD)

## 섹션별 가이드

### 배경

"왜 이걸 만드는지"를 1-3문장으로. Claude Code가 요구 사항을 해석할 때 방향을 잡아주는 역할. 요구 사항만으로 맥락이 명확하면 생략한다.

### 요구 사항

개발해야 할 것을 도메인 용어로 기술한다.
- 추가/변경을 구분하면 읽기 좋다
- 구현 세부사항(테이블명, API 경로)은 쓰지 않는다 — 코드에서 결정할 내용

### 정책

요구 사항의 "무엇을"과 분리하여 "어떤 규칙으로"를 정리한다. 해당 사항이 없으면 섹션 자체를 생략.

### 작업 목록

체크리스트로 진행 상황을 추적한다.
- `- [ ]` 미완료, `- [x]` 완료
- 항목은 "끝났는지 아닌지 판단 가능한 단위"
- 새 작업이 생기면 추가, 불필요해지면 제거
- 순서는 대략적인 작업 순서를 따르되 엄격하지 않아도 됨

### 관련 문서

관련 daybook 섹션 파일의 링크를 관리한다. lookup 커맨드가 이 섹션을 읽어 daybook 내용을 조회한다.

- 링크 형식: `[[YYYY/MM/DD/주제]]`
- 중복 링크 방지

## 용어 규칙

프로젝트 문서는 기획·개발 관점의 문서다. 이름은 언제든 바뀔 수 있으므로 의도와 역할로 서술한다.

**기술 용어 지양**
- 테이블명, 클래스명, 컬럼명 등 코드 수준의 용어는 사용하지 않는다
- 요구 사항: 도메인 용어로 작성
- 작업 목록: 도메인 관점으로 작성

**기능명·시각 표현 지양**
- 기능명이나 시각적 표현은 바뀔 수 있으므로 역할 기반 용어를 사용한다
- 요구 사항은 기획 원문이므로 기능명을 그대로 유지해도 된다
- 작업 목록에서는 역할로 치환한다
