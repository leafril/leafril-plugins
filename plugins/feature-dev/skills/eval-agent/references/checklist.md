# 에이전트 구조 평가 체크리스트

공식 sub-agents 문서 + skill best practices 기반. 각 항목을 O/X/해당없음으로 판정한다.

## description

| # | 항목 | 판정 기준 |
|---|------|-----------|
| 1 | "무엇을 하는지" 포함 | 에이전트의 기능이 명확히 기술되어 있는가 |
| 2 | "언제 위임하는지" 포함 | Claude가 이 에이전트에 위임해야 하는 상황/조건이 기술되어 있는가. "when to delegate" 관점. |
| 3 | 구체적 키워드 포함 | 모호한 표현("helps with code") 없이 도메인 핵심 키워드가 있는가 |
| 4 | 3인칭으로 작성 | 1인칭("I review..."), 2인칭("You can use...") 미사용. orchestrator가 description을 읽고 위임 대상을 선택하므로, 1인칭은 "I"가 누구인지 모호해져 discovery가 깨진다 |
| 5 | 1024자 이하 | 최대 길이를 초과하지 않는가. 초과 시 플랫폼에서 자동 절삭되어 discovery가 불완전해진다 |

## frontmatter

| # | 항목 | 판정 기준 |
|---|------|-----------|
| 6 | name 규칙 준수 | 소문자+하이픈, 고유 식별자인가 |
| 7 | tools 최소 권한 | 에이전트 역할에 필요한 도구만 명시했는가. 불필요한 도구가 포함되어 있지 않은가. (tools ↔ 제약 정합성은 #19에서 별도 판정) |
| 8 | tools vs disallowedTools 충돌 없음 | 양쪽에 같은 도구가 있으면 제거되므로 의도적인지 확인. 둘 다 지정한 경우 의미가 명확한가 |
| 9 | model 선택 적절 | 작업 복잡도에 맞는 모델인가. 단순 조회에 opus, 복잡 추론에 haiku이면 X. 생략(inherit)이면 해당없음 |
| 10 | maxTurns 존재 | 설정되어 있는가. 미설정이면 무한 루프 위험 |
| 11 | maxTurns 적정성 | 값이 작업 복잡도에 비례하는가 (간단한 분석 3-10, 복잡한 다단계 20-30). 판정 시 근거를 비고에 기술 |
| 12 | permissionMode 안전성 | bypassPermissions 사용 시 정당한 근거가 있는가. 플러그인 에이전트에 permissionMode가 있으면 X (무시됨) |

## system prompt (body)

| # | 항목 | 판정 기준 |
|---|------|-----------|
| 13 | 컨텍스트 자립적 | 부모 대화 이력 없이 이 prompt만으로 역할·절차·제약을 이해할 수 있는가. "호출자의 context를 모른다" 같은 명시적 선언이 있으면 가점 |
| 14 | 입력 명세 | 에이전트가 받는 입력(프롬프트 파라미터)이 무엇인지 정의되어 있는가 |
| 15 | 출력 형식 정의 | 다른 에이전트/스킬이 파싱해야 하는 출력이 있다면, 형식이 명확히 정의되어 있는가 (예: RESULTS:, VERDICT: 등). 독립 실행 에이전트면 해당없음 |
| 16 | 제약/금지 사항 명시 | 에이전트가 하지 말아야 할 행동이 구체적으로 나열되어 있는가. "주의한다" 같은 모호한 표현이 아닌 "X하지 않는다" 형태 |
| 17 | 간결함 | Claude가 이미 아는 내용을 장황하게 설명하지 않는가. 불필요한 배경 설명이 없는가 |
| 18 | body 길이 적정 | system prompt가 500줄 이하인가. 초과 시 instruction-following이 저하된다 |
| 19 | 시간 의존 정보 없음 | "2025년 이후엔..." 같은 표현이 없는가 |
| 20 | 용어 일관성 | 같은 개념에 다른 용어를 혼용하지 않는가 (예: "criterion"과 "기준"을 혼용) |

## 안전 경계

| # | 항목 | 판정 기준 |
|---|------|-----------|
| 21 | tools ↔ 제약 정합성 | system prompt의 제약("코드를 수정하지 않는다")과 tools 목록(Write/Edit 포함 여부)이 일치하는가. 제약은 있는데 도구가 허용되면 X |
| 22 | 서브에이전트 생성 제한 | Agent 도구가 허용되어 있다면, 생성 가능한 서브에이전트 타입이 제한되어 있는가 (`Agent(type1, type2)` 형태). 무제한이면 정당한 이유가 있는가 |
| 23 | 파괴적 작업 안전장치 | Bash로 파괴적 명령(rm, DROP, reset --hard 등)을 실행할 수 있는 에이전트라면, PreToolUse hook이나 prompt 내 제약이 있는가 |

## 통합 설정 (해당 시)

| # | 항목 | 판정 기준 |
|---|------|-----------|
| 24 | skills 프리로드 적절 | skills 필드에 나열된 스킬이 에이전트 역할에 실제로 필요한가. 불필요한 스킬이 컨텍스트를 낭비하고 있지 않은가 |
| 25 | memory scope 적절 | memory가 설정되어 있다면 scope(user/project/local)가 에이전트 용도에 맞는가. 프로젝트 특화 지식인데 user scope이면 X |
| 26 | mcpServers 최소화 | 불필요한 MCP 서버가 포함되어 있지 않은가. 플러그인 에이전트에 mcpServers가 있으면 X (무시됨) |
| 27 | hooks 유효성 | hooks가 정의되어 있다면 matcher와 command가 유효한가. 플러그인 에이전트에 hooks가 있으면 X (무시됨) |
| 28 | 플러그인 제약 인지 | 플러그인 에이전트인 경우: hooks, mcpServers, permissionMode가 모두 없는가 (있으면 무시됨을 사용자에게 경고) |
