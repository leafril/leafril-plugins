# Daybook 작성 규칙

## 파일 구조

### 일별 디렉터리
- 패턴: `daybook/{YYYY}/{MM}/{DD}/`
- 예시: `daybook/2026/02/26/`

### 섹션 파일
- 패턴: `daybook/{YYYY}/{MM}/{DD}/{주제}.md`
- 예시: `daybook/2026/02/26/API 응답 포맷 결정.md`
- 역할: 하나의 주제에 대한 기록. 일별 디렉터리 안에 주제별로 파일을 생성한다.

### 링크 규칙
- 링크 시 연도/월/일 경로를 명시한다
- 예시: `[[2026/02/26/API 응답 포맷 결정]]`

## 섹션 파일 템플릿

```markdown
---
created: YYYY-MM-DD
---

> [!note] Summary
> ...

서술부. 자유 형식으로 기록.
```

### frontmatter 필드

**필수:**
- `created`: 작성 날짜

**선택 (해당할 때만):**
- `organization`: 조직명
- `project`: 프로젝트(리포)명
- `task`: 관련 문서 링크. `"[[문서명]]"` 형식 (YAML에서 `[[]]`는 따옴표 필수)
- `tags`: 기술/개념 태그 목록

**계층 관계:** `organization` > `project` > `task` — 상위 없이 하위만 쓰지 않는다

### 예시 — 프로젝트 관련 작업

```markdown
---
created: 2026-02-26
organization: acme
project: user-api
task: "[[인증-흐름-개선]]"
---

> [!note] Summary
> ...

서술부.
```

### 예시 — 일반 기록

```markdown
---
created: 2026-02-27
tags:
  - testing
---

> [!note] Summary
> ...

서술부.
```

### 태그 규칙
- 구조화된 메타데이터(폴더 경로, frontmatter 필드)와 중복되는 태그는 달지 않는다
  - 예: `project: user-api`가 있는데 `user-api` 태그 ❌
- daybook 섹션 파일은 날짜 폴더 안에 있어 주제 분류가 없으므로, 주제 태그가 유일한 구조화된 분류 수단이다
  - 예: `로그 레벨 설정 기준 정리.md` → `tags: - logging` ✅
- 인라인 태그(`#태그`)는 사용하지 않는다 — frontmatter `tags`로 일원화

## Summary 콜아웃

모든 섹션 파일의 상단에 붙인다. 콜아웃으로 요약을 먼저, 아래 서술부에서 맥락을 기록한다.

핵심 레이블:
- **배경** — 왜 이 기록이 필요한가, 어떤 상황인가
- **결론** — 무엇을 알게 되었나, 어떻게 결정했나

나머지는 내용에 맞게 자유롭게 쓴다. 예: 문제, 원인, 해결, 검토, 선택, 적용, 느낀점 등.

레이블 내용은 단문이면 그대로, 복수 항목이면 불릿으로 정리한다.

```markdown
> [!note] Summary
> **배경**
> ...
>
> **결론**
> ...

서술부.
```

```markdown

> - [ ] 또 다른 작업
```

## 작성 원칙 (Dave Thomas 스타일)
1. **구조보다 내용**: 형식에 얽매이지 않고 자유롭게
2. **나중의 나를 위해**: "그때 어떻게 했지?" 찾아볼 수 있도록
3. **실수도 기록**: 실수와 피하는 방법도 중요한 기록
4. **같은 맥락은 한 파일에**: 오늘 같은 주제의 파일이 있으면 덧붙인다. 다른 날의 파일은 수정하지 않고 새 파일을 만든다.

## 서술과 코드 분리 원칙

daybook은 "왜 이렇게 했는지"를 미래의 나에게 전달하는 문서다. 코드 구현은 바뀌지만 판단의 이유는 바뀌지 않는다. 서술에 클래스명이나 메서드명이 섞이면, 리팩토링 후 그 이름이 사라졌을 때 기록이 의미를 잃는다.

**핵심: 서술은 "무엇을 왜"로, 코드는 코드 블록으로.**

- 서술에서는 도메인 개념이나 행동/의도로 표현한다. 구체적인 코드 이름은 쓰지 않는다.
- 코드가 필요하면 코드 블록 안에서만 사용한다.
- 테이블명, 컬럼명 같은 구현 디테일도 서술이 아닌 코드 블록이나 표에서만 언급한다.

**나쁜 예** — 서술에 구현이 섞임:
```markdown
이벤트 트래커에서 `trackEvent()`를 호출할 때 `EventProperties`를
직렬화해야 해서 `toMap()`을 추가했다.
```

**좋은 예** — 서술은 개념으로, 구현은 분리:
````markdown
이벤트 트래커가 속성 객체를 직렬화할 수 없어서 변환 메서드를 추가했다.

```kotlin
fun toMap(): Map<String, Any> { ... }
```
````

**판단 기준**: "이 코드가 리팩토링되어 이름이 바뀌어도 이 문장이 여전히 의미가 있는가?"
답이 No라면 서술에서 코드 이름을 빼고 개념으로 대체한다.
