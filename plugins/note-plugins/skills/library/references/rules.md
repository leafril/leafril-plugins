# Library 작성 규칙

## 목차
- 핵심 원칙 — 80% 커버하는 핵심만 담는다
- 파일 구조 — 경로, 디렉터리, 파일명 규칙
- 문서 템플릿 — frontmatter, 한 줄 요약, 섹션 구조
- 본문 작성 원칙 — 핵심 20%, 문제 중심 구조, 참조 섹션

---

## 핵심 원칙

**"이것만 알면 80% 상황을 커버한다"는 핵심만 담는다.**

- 공식 문서는 모든 사람을 위한 완전한 정보. library는 내가 몰랐던 것의 압축본.
- 문서에 모든 내용을 담지 않는다. 핵심 20%만 적고, 나머지는 참조 링크로 넘긴다.
- 문제 중심으로 쓴다: 이게 왜 필요한가 → 핵심 → 함정/교훈.

## 파일 구조

### 경로

디렉터리가 분류 역할을 한다. 두 가지 유형이 있다:

1. **기술/도구 디렉터리** — 특정 기술에 속하는 개념
   - 패턴: `library/{기술명}/{개념명}.md`
   - 예: `library/JPA/@Modifying.md`, `library/PostgreSQL/VACUUM.md`, `library/Docker/멀티스테이지 빌드.md`

2. **도메인 디렉터리** — 특정 기술에 묶이지 않는 범용 개념
   - 패턴: `library/{도메인}/{개념명}.md`
   - 예: `library/design/YAGNI.md`, `library/network/커넥션 풀.md`, `library/web/MIME 타입.md`

### 디렉터리 이름

- **기술 디렉터리**: 공식 이름 그대로 (예: `JPA`, `PostgreSQL`, `Docker`, `AWS`, `Git`)
- **도메인 디렉터리**: 소문자 영어 (예: `design`, `network`, `web`)

기존 디렉터리 목록을 먼저 확인하고, 해당하는 것이 있으면 사용한다.
없으면 새 디렉터리를 생성한다.

### 디렉터리 유형 판단

- 개념이 특정 기술/프레임워크/서비스에 속한다 → 기술 디렉터리
- 여러 기술에 걸치는 범용 개념이다 → 도메인 디렉터리

### 생태계 그룹핑

같은 생태계에 속하는 기술은 개별 디렉터리를 만들지 않고 **생태계 디렉터리**에 묶는다. 파일이 충분히 쌓여서(5개 이상) 독립할 필요가 생기기 전까지는 그룹으로 유지한다.

- **JavaScript/** — Node.js, Next.js, React, TanStack Query, Lottie 등 JS 생태계 전체
- **Linux/** — SSH, 셸 명령어 등 Linux 도구 포함

판단 기준: 개별 기술의 파일이 1~2개뿐이면 생태계 디렉터리에 둔다. 5개 이상 쌓이면 독립 디렉터리로 분리를 검토한다.

### 파일명

디렉터리가 기술 맥락을 제공하므로, 파일명에 기술 접두사를 붙이지 않는다.
- 좋은 예: `JPA/@Modifying.md`, `PostgreSQL/timestamptz.md`
- 나쁜 예: `JPA/JPA-Modifying.md`, `PostgreSQL/PostgreSQL-timestamptz.md`

파일명은 개념의 공식 이름 또는 가장 널리 쓰이는 이름:
- 영어 공식 이름이 있으면 영어로 (예: `VACUUM.md`, `S3.md`)
- 한국어가 자연스러운 개념은 한국어로 (예: `커넥션 풀.md`, `멀티스테이지 빌드.md`)

## 문서 템플릿

```markdown
---
aliases:
  - {별칭1}
---
{한 줄 요약. 역할/효과 중심으로 쉽게.}

## 왜 필요한가

어떤 문제를 해결하는지.

## 핵심

이것만 알면 80% 상황을 커버하는 내용.

## 함정

실제로 빠지기 쉬운 실수, 주의사항.

## Q&A

이해도 확인 과정에서 나온 질문과 답변. "왜 이렇게 되는 거지?"를 기록.

### 질문 제목 (헷갈렸던 포인트를 질문으로)
답변. 핵심만 간결하게.

## 다음 학습

이 문서에서 얕게 언급했거나 다루지 않은 관련 개념.

- **개념명** — 왜 다음에 배우면 좋은지 한 줄 이유

## 참조
- [공식 문서 제목](URL)
```

### frontmatter

**aliases만 사용한다.** tags, category 등 다른 분류 필드는 쓰지 않는다 — 디렉터리가 분류를 대신한다.

aliases 규칙:
- 영어 문서라면 한국어 별칭, 한국어 문서라면 영어 별칭을 포함
- 약어의 풀네임 또는 풀네임의 약어 포함
- 예: `YAGNI.md` → `aliases: [You Aren't Gonna Need It]`
- 예: `커넥션 풀.md` → `aliases: [Connection Pool]`
- 별칭이 없는 경우 frontmatter 자체를 생략해도 됨

## 한 줄 요약

frontmatter 바로 아래, 첫 번째 섹션 전에 위치한다.
**역할/효과 중심**으로, 처음 배우는 사람도 이해할 수 있게 쉽게 쓴다.

- 전문 용어를 최소화하고, 무엇을 해결하는지에 초점
- 좋은 예: `삭제/수정 후 남는 불필요한 데이터를 정리해서 디스크를 확보하는 명령.`
- 좋은 예: `"지금 필요하지 않은 것은 만들지 마라"`
- 나쁜 예: `MVCC가 남기는 dead tuple을 정리하여 공간을 회수하는 유지보수 명령.` (전문 용어가 정의 없이 등장)
- 나쁜 예: `커넥션 풀에 대한 정리.` (아무 정보 없음)

## 본문 작성 원칙

### 핵심 20%만 담는다

- 실무에서 80% 상황을 커버하는 내용만 적는다
- 모든 옵션, 모든 설정을 나열하지 않는다 — 그건 공식 문서의 역할
- 상세한 동작 원리, 전체 파라미터 목록 등은 참조 링크로 대체
- 코드 예시는 최소한으로, 핵심을 보여주는 정도만
- 추상적인 설명에는 구체적인 시나리오 예시를 붙인다 — "왜?"나 "어떻게?"가 말로만 설명되면 와닿지 않는다. 단계별 상태 변화를 보여주는 방식이 가장 효과적이다 (예: "첫 커밋 → 5개 객체 생성, README만 수정 → 3개 추가, 2개 재사용"). 비교 다이어그램, before/after도 활용한다
- 다이어그램이나 흐름도는 Mermaid를 사용한다
- **HTML/CSS 개념은 데모 파일을 만든다** — ASCII 다이어그램 대신 실제 동작하는 `.html` 파일로 예시를 제공한다. 데모 파일은 참조하는 문서와 같은 디렉토리에 둔다. 문서에서는 `[데모 파일명](파일명.html)` 형식으로 링크한다

### 문제 중심 구조

섹션은 필요한 것만 골라 사용한다:
- **왜 필요한가** — 이 개념이 해결하는 문제 (어떤 상황에서 이게 없으면 뭐가 문제인지)
- **핵심** — 이것만 알면 되는 내용
- **함정** — 실제로 빠지기 쉬운 실수, 내가 겪은 교훈 ("하지 마라" 중심)
- **Q&A** — 이해도 확인에서 나온 질문과 답변 ("왜 이렇게 되는 거지?" 중심)
- **다음 학습** — 이 문서에서 얕게 언급한 관련 개념과 학습 이유
- **참조** — 공식 문서 링크 (더 알고 싶으면 여기로)

모든 섹션을 채울 필요 없다. 개념의 성격에 맞는 섹션만 사용한다.

**함정 vs Q&A 구분:**
- 함정: 리서치에서 나온 알려진 실수/주의사항 (예: "none 알고리즘 공격")
- Q&A: 이해도 확인에서 사용자가 실제로 헷갈린 부분 (예: "RS256에서 각 서비스가 다른 키를 갖는 거 아닌가?")
- 둘이 겹치면 함정에 유지하고 Q&A에는 넣지 않는다

### 참조 섹션

`## 참조` 섹션에 공식 문서 등 외부 레퍼런스를 기록한다.
- 마크다운 링크 형식: `[표시 텍스트](URL)`
- 공식 문서, 스펙, 신뢰할 수 있는 기술 블로그 등
- 내부 위키링크(`[[노트명]]`)는 개념적으로 강하게 연결된 경우에만 함께 사용
- 참조할 외부 문서가 없으면 섹션 자체를 생략
