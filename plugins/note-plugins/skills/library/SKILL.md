# Library

개발 중 새로 알게 된 개념, 기술, 패턴을 지식 노트로 정리하는 스킬.

핵심 가치: **"이거 뭐였지?"를 3초 안에 찾아볼 수 있는 레퍼런스 만들기.**

daybook이 "그때 어떻게 했지?"를 기록한다면, library는 "이 개념이 뭐지?"에 답한다.
daybook은 경험 중심, library는 개념 중심.

## 설정

`~/.claude/note-plugins.json`에서 `noteRoot`를 읽는다. 파일이 없거나 `noteRoot`가 없으면 사용자에게 물어보고 저장한다.

## 작성 규칙

파일 경로, 템플릿, frontmatter, 디렉터리 규칙은 [references/rules.md](references/rules.md)에 정의되어 있다.
실행 시 이 파일을 읽고 그에 맞춰 작성한다.

## 워크플로우

### Phase 1: 주제 파악

1. `$ARGUMENTS`에서 핵심 개념을 파악한다
2. 기존 library에 같은 주제의 문서가 있는지 확인한다
   - 있으면 → 기존 문서를 읽고, 새 내용을 추가/수정
   - 없으면 → 새 문서 생성

### Phase 2: 리서치

사용자가 전달한 내용을 기반으로, 부족하거나 불정확한 부분을 웹 검색으로 보강한다.

- 사용자가 전달한 핵심 맥락은 유지하되, 정확성을 확인한다
- 사용자가 언급하지 않은 중요한 옵션이나 주의사항이 있으면 추가한다
- 과도하게 확장하지 않는다 — 사용자가 필요로 하는 범위에 집중

### Phase 3: 작성

references/rules.md의 템플릿과 규칙에 따라 문서를 작성한다.

**새 문서 생성 시:**
- 디렉터리를 결정한다: 특정 기술에 속하면 기술명 디렉터리, 범용 개념이면 도메인 디렉터리
- 기존 디렉터리가 없으면 생성
- 템플릿에 따라 aliases + 본문 작성

**기존 문서 수정 시:**
- 새 섹션 추가 또는 기존 내용 보강
- 기존 구조를 존중하면서 자연스럽게 통합

### Phase 3.5: 자가 검증

문서 작성 직후, rules.md의 핵심 규칙을 대조하여 검증한다. 작성한 문서를 다시 읽고 아래를 하나씩 확인한다:

1. 문서에 등장하는 용어 중 독자가 모를 수 있는 것이 있는가? → 있으면 용어 정리 표 추가
2. "왜 필요한가"부터 코드 예제까지 하나의 시나리오로 관통하는가?
3. 코드에 각 줄이 뭘 하는지 주석이 있는가?
4. 코드 아래에 흐름을 풀어서 설명하는 산문이 있는가?
5. 함정 섹션에 "왜 이런 일이 생기는가" 원인이 있는가?
6. 한 줄 요약에 전문 용어가 정의 없이 등장하지 않는가?
7. 서술에서 기술 용어를 빼도 의미가 통하는가? → 통하면 빼고 역할/행동으로 대체

위반 항목이 있으면 문서를 수정하고, 수정한 항목을 다시 확인한다. 모두 통과할 때까지 반복한 뒤 다음 Phase로 넘어간다.

### Phase 4: 이해도 확인

> `/library`로 수동 호출한 경우에만 실행한다. 자동 호출된 경우 Phase 4~5를 생략하고 파일 경로 + 한 줄 요약만 보고한다.

문서 작성 직후, 사용자가 내용을 제대로 이해했는지 단계적으로 확인한다.

**질문 수는 문서 복잡도에 비례한다:**
- 단순 개념 (YAGNI 등) → 1~2개
- 보통 개념 (Presigned URL 등) → 2~3개
- 복잡한 개념 (Kafka Partition 등) → 3~5개

**단계적 심화:**
1. **핵심 확인** — 문서의 핵심을 이해했는지 직접적인 질문 (예: "partition이 왜 필요한가요?")
2. **적용 확인** — 상황을 주고 판단을 묻기 (예: "주문 이벤트의 순서가 중요하다면 key를 어떻게 설정할까요?")
3. **함정 확인** — 실수하기 쉬운 부분을 시나리오로 (예: "partition 3개에 consumer 5개면 어떻게 될까요?")

답변이 부족하거나 틀리면 → 해당 부분을 보충 설명하고 → **문서에도 보충 내용을 반영**한다 → 다른 각도로 다시 확인한다.
정확히 답하면 → 다음 단계로 넘어가거나 Phase 5로 진행한다.

**문서 업데이트 원칙:**
- 이해도 확인 과정에서 사용자가 모르거나 헷갈린 부분이 드러나면, 그건 문서의 설명이 부족했다는 신호다. 보충 설명한 내용(예시, 연결고리 등)을 문서에 추가하여 다음에 다시 볼 때도 이해할 수 있게 한다.
- 사용자가 헷갈린 질문과 답변은 `## Q&A` 섹션에 기록한다. 질문 제목은 사용자가 실제로 헷갈린 포인트를 질문 형태로 쓰고, 답변은 핵심만 간결하게 적는다.
- 함정과 겹치는 내용은 함정에 유지하고 Q&A에는 넣지 않는다.

### Phase 5: 다음 학습 추천

이해도 확인 후, 이 문서에서 의도적으로 다루지 않았거나 얕게만 언급한 개념 중 다음에 학습하면 좋을 것들을 2~3개 제안한다.

- 문서 본문에서 설명 없이 등장한 용어나, "함정"에서 언급만 한 개념을 우선 추천
- 각 추천에 왜 다음에 배우면 좋은지 한 줄 이유를 붙인다
- AskUserQuestion의 `options` 파라미터로 선택지를 제시한다
- **다음 학습 추천 내용을 문서의 `## 다음 학습` 섹션에도 기록한다** — 나중에 문서를 다시 볼 때 학습 경로를 이어갈 수 있게

## 판단 기준

### 새 문서 vs 기존 문서 수정

- 같은 개념에 대한 새로운 측면 → 기존 문서에 섹션 추가
- 완전히 다른 개념 → 새 문서
- 관련이 있지만 독립적인 개념 → 새 문서 + `## 관련` 섹션에서 상호 링크

### 자동 감지 기준

대화 중 아래 패턴이 보이면 library 문서 작성을 제안한다:
- 사용자가 "이게 뭐야?", "처음 알았다", "몰랐는데" 같은 반응을 보일 때
- 개념 설명 후 "정리해두면 좋겠다" 같은 의도가 보일 때
- 단, 이미 잘 알고 있는 개념에 대한 단순 질문은 제외

## 대화 방식

- 메타데이터를 묻는 폼식 질문은 하지 않는다
- 디렉터리, aliases, 파일명은 스스로 판단한다
- 내용이 불충분하면 개념의 맥락을 끌어내는 질문을 한다 ("어떤 상황에서 이걸 알게 됐어요?", "어떤 부분이 헷갈렸나요?")
- 리서치로 보강한 내용 중 사용자의 원래 이해와 다른 부분이 있으면 알려준다
