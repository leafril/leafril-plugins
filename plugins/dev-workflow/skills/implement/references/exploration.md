## 목적

이 기능을 **새로** 만드는 게 아니라 **기존 패턴에 편입**시키기 위함. 컨벤션 문서(`.claude/rules/*.md`)가 선언적 규칙을 주고, **실제 코드는 그 규칙이 어떻게 구체화됐는지**를 보여준다. 둘 다 필요.

## 가설 주도 탐색

"뭐가 있나 보자"로 여는 파일은 컨텍스트 낭비. 탐색 전 **가설**을 먼저 세운다.

예:
- "이 기능은 외부 HTTP API를 호출한다 → 기존 HTTP 클라이언트 패턴이 있을 것"
- "이 기능은 DB에 쓴다 → 기존 영속 엔티티 + Repository 패턴을 따를 것"
- "이 기능은 테스트가 필요하다 → 유사 기존 기능의 테스트 구조를 따를 것"

가설을 **한 번의 탐색으로 확인**한다. 확인되면 멈춤. 확인 안 되면 가설 수정.

## 탐색 대상 체크리스트

가설에 따라 다음 중 해당되는 것만.

**컨벤션·규칙 (선언)**
| 무엇 | 어떻게 | 왜 |
|---|---|---|
| 프로젝트 가이드 | `CLAUDE.md`, `.claude/rules/*.md` | 아키텍처·네이밍·컨벤션 명시 규칙 |
| 커밋 컨벤션 | `git log --oneline -20` | 메시지 포맷·스타일·티켓 prefix 관용 |
| 브랜치 네이밍 | `git branch -a` 몇 개 샘플 | prefix·kebab-case 등 |

**기존 코드 (구현)**
| 무엇 | 어떻게 | 왜 |
|---|---|---|
| 같은 도메인의 유사 기능 | Glob/Grep으로 `**/*{유사 도메인}*` | 레이어 구조·DI·네이밍 모방 |
| 같은 외부 시스템 호출 (HTTP·S3·Kafka 등) | 기존 클라이언트/어댑터 grep | 타임아웃·재시도·에러 매핑 관용 |
| 같은 영속 패턴 | 기존 엔티티/Repository 파일 | 스키마·soft delete 관용 |
| 같은 테스트 유형 | 기존 테스트 파일 | Base 클래스·fixture·mock 전략 |
| rules 문서와 코드의 gap | 둘을 대조 | 문서에 없는 암묵 컨벤션 발견 |

**변경 이력 (맥락)**
| 무엇 | 어떻게 | 왜 |
|---|---|---|
| 건드릴 파일의 최근 변경 | `git log -10 -- <path>`, `git blame -L ...` | "왜 이렇게 돼있나" 설계 의도 |
| 관련 최근 PR | `gh pr list --search "..."`, `gh pr view <num>` | 같은 도메인에서 최근 결정 |
| 진행 중 WIP | `git log origin/main..HEAD`, `gh pr list --author @me` | 동시 진행 작업과 충돌 가능성 |
| 유관 progress plan | `ls progress/` → 관련 plan read | 선행·병렬 기능 간 계약 |
| 과거 유사 feature의 커밋 | `git log --grep "<키워드>"` | 당시 어떻게 쪼개서 구현했나 |

## 탐색 깊이 기준

- **읽는 파일 수 ≤ 5**. 이 이상이면 가설이 너무 넓음
- **파일당 Read 범위**: 대부분 offset 0부터 100줄 정도면 패턴 드러남. 전체 파일 필요 드묾
- **Grep 쿼리 ≤ 5**. 더 필요하면 가설 쪼개서 Agent(Explore) 위임 고려

## 탐색 결과 사용

- **steps 초안에 반영**: "기존 X 패턴을 따른다"는 식. 파일 경로를 step에 박지 말고 머릿속에 유지
- **사용자 보고에 1~2줄**: "기존 `{유사 기능}` 패턴을 따를 예정" 정도. 탐색 덤프 금지
- **step 구현 시**: 특정된 1~2개 모범 파일을 옆에 두고 그대로 모방

## 탐색을 멈추는 기준

다음 중 하나라도 충족하면 즉시 멈추고 다음 단계로:

- 가설이 확인됐다 (이 패턴을 따라 구현하면 된다는 확신)
- 기존에 유사 패턴이 **없다**는 결론 (rules 문서만 따라 새로 설계)
- 5개 파일·5개 grep을 썼는데도 감이 안 옴 → 사용자에게 "탐색 한계, 어떻게 접근할지 질문"

## 하지 말 것

- 관련성 불분명한 코드 폭넓게 읽기
- "있어 보이니까" 파일 manifest로 만들기 (plan은 path를 담지 않음)
- 컨벤션 문서를 다시 찾아 읽지 않고 추측
- 탐색 중 발견한 기존 버그·개선 기회 기록 (이 기능 범위 밖 — 별도 티켓)
