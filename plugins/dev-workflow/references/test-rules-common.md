# 테스트 작성 규칙 (공통)

언어·프레임워크 무관 보편 원칙. 언어별 구체 적용은 `test-rules-{lang}.md`를 참조한다.

## 목차
- §1 코드 유형 분류 (2×2 매트릭스) — 도메인 모델, 컨트롤러, 사소한 코드, 과도하게 복잡한 코드
- §2 테스트하지 않는 것 — 사소한 코드, 프레임워크 보장 동작, 관리 의존성 단독 테스트, 동일 패턴 반복
- §3 테스트 스타일 우선순위 — 출력 > 상태 > 통신 기반
- §4 통합 테스트 수 가이드 — happy path 하나 + 비즈니스 규칙 단위
- §5 mock 규칙 — managed는 실제 사용, unmanaged만 mock
- §6 구조 규칙 — 이름, GWT, 테스트당 하나의 행위, fixture
- §7 자가 평가 체크리스트 — 이름, assert, mock, 중복, 데이터 충분성 검증

---

## 1. 코드 유형 분류 (2×2 매트릭스)

테스트 대상을 **협력자 수**(복잡도)와 **도메인 중요도** 두 축으로 분류한다.

|  | 도메인 중요도 높음 | 도메인 중요도 낮음 |
|--|--|--|
| **협력자 적음** | **도메인 모델** — 단위 테스트 집중 | **사소한 코드** — 테스트 안 함 |
| **협력자 많음** | **과도하게 복잡한 코드** — 리팩터링 먼저 | **컨트롤러** — 통합 테스트 happy path만 |

### 유형별 테스트 전략

| 유형 | 전략 | 예시 개념 |
|------|------|------|
| 도메인 모델 | 단위 테스트로 모든 분기·경곗값 커버 | 계산 로직, 필터링 규칙, 값 변환/매핑 |
| 컨트롤러 | 통합 테스트로 가장 긴 happy path 하나 | 외부 시스템 호출 → 변환 → 저장 흐름 |
| 사소한 코드 | 테스트하지 않음 | 필드 매핑 팩토리, 단순 위임, 설정 |
| 과도하게 복잡한 코드 | 도메인 모델과 컨트롤러로 분리 후 각각 테스트 | 비즈니스 로직이 협력자와 얽혀 있는 경우 |

### 과도하게 복잡한 코드 감지

아래 조건을 **모두** 만족하면 과도하게 복잡한 코드로 분류한다:

1. 단위 테스트로 검증해야 할 도메인 로직(조건 분기, 계산, 필터링)이 있다
2. 그 로직이 협력자(외부 저장소, 외부 API 등)와 얽혀 있어 협력자 없이 테스트할 수 없다

이는 도메인 로직을 별도 도메인 모델로 추출하라는 신호다. 추출 후 도메인 모델은 단위 테스트, 남은 컨트롤러 계층은 통합 테스트 happy path 한 건으로 분류한다.

## 2. 테스트하지 않는 것

항상 통과하는 테스트는 회귀를 감지하지 못한다. 테스트의 가치는 "코드가 깨졌을 때 실패하는가"에 있으므로, 깨질 수 없는 테스트는 유지 비용만 발생시킨다.

다음에 해당하면 테스트를 작성하지 않는다. **기존 테스트에서 발견하면 삭제를 권고한다.**

### 사소한 코드
- 필드 매핑만 하는 팩토리/변환 함수 (null → 기본값 같은 단순 분기 1~2개 포함해도 사소한 코드)
- 다른 메서드로 단순 위임하는 코드
- 접근자(getter/setter), 표현용 메서드(toString류)

### 프레임워크·언어·런타임이 보장하는 동작
- 예외 전파 (중간에 잡지 않으면 당연히 전파됨)
- 표준 라이브러리의 문서화된 동작 (예: 사전 자료구조의 없는 키 조회는 빈 값 반환)
- 저장소·ORM·트랜잭션 미들웨어가 문서로 보장하는 동작
- 프레임워크가 주입·생성해 주는 객체의 기본 lifecycle

> 원칙: "이 assert는 프로덕션 코드의 로직이 깨졌을 때 실패하는가, 아니면 프레임워크가 망가져야만 실패하는가?"를 물어본다. 후자라면 삭제 대상이다.

### 관리 의존성 단독 테스트
- 인메모리 저장소, 직렬화 어댑터 등을 별도 단위 테스트로 다루지 않는다
- 이들은 컨트롤러 통합 테스트에서 실제 사용되며 자연스럽게 커버된다

### 동일 패턴 반복
- 시간 단위 집계 테스트가 있으면 일/주/월 동일 패턴 → 추가하지 않음
- A 유형 동기화 테스트가 있으면 B 유형도 동일 패턴 → 추가하지 않음
- 저장/교체 테스트가 이미 있으면 같은 패턴의 다른 필드 조합은 추가하지 않음

## 3. 테스트 스타일 우선순위

리팩터링 내성이 높은 순서대로 선호한다. 내부 구현을 바꿨을 뿐인데 테스트가 깨지면(거짓 양성) 테스트 신뢰도가 떨어진다. 출력 기반은 반환값만 보므로 내부 변경에 면역이고, 통신 기반은 호출 순서까지 검증하므로 가장 깨지기 쉽다.

| 순위 | 스타일 | 설명 | 사용 시점 |
|------|--------|------|-----------|
| 1 | **출력 기반** | 반환값만 검증 | 도메인 모델의 순수 함수 |
| 2 | **상태 기반** | 실행 후 저장소/객체 상태 검증 | managed dependency 관련 |
| 3 | **통신 기반** | mock 호출 여부 검증 | unmanaged dependency로 나가는 **명령**에만 |

출력 기반으로 쓸 수 있으면 출력 기반으로, 상태 기반으로 쓸 수 있으면 상태 기반으로 내려와라. 통신 기반은 마지막 수단이다.

---

## 4. 통합 테스트 수 가이드

### 가장 긴 happy path 하나

통합 테스트는 실행 비용(시간, 외부 리소스, 환경 초기화)이 높고 실패 원인 추적이 어렵다. 수를 최소화하되, 프로세스 외부 의존성과의 상호작용이 실제로 동작하는지는 반드시 확인해야 한다. "가장 긴 happy path 하나"는 이 균형점이다.

happy path란 **하나의 비즈니스 시나리오가 성공적으로 실행되는 경로**다. "가장 긴" happy path는 그 시나리오 안에서 프로세스 외부 의존성(저장소, 외부 API 등)과의 상호작용을 가장 많이 거치는 경로를 의미한다.

- 하나의 비즈니스 시나리오 = 하나의 공개 진입점 호출이 트리거하는 흐름
- 독립적인 진입점을 인위적으로 연결하여 긴 경로를 만들지 않는다
- 비즈니스 시나리오별로 하나의 happy path 테스트를 둔다

### 비즈니스 규칙 테스트는 추가 가능

컨트롤러 수준에서 "교체", "보존", "멱등성" 같은 비즈니스 규칙이 있으면 해당 규칙 단위로 추가 테스트를 둘 수 있다. 단, 도메인 모델 단위 테스트에서 커버 가능한 분기는 통합 테스트에서 중복하지 않는다.

### 같은 패턴의 다른 수준 반복 금지

시간별 ↔ 일별, 유형 A ↔ 유형 B처럼 구조가 동일한 코드는 하나만 테스트한다.

## 5. mock 규칙

| 의존성 종류 | 예시 개념 | mock 여부 |
|------------|------|-----------|
| **managed** (내가 제어) | 프로젝트 전용 저장소, 인메모리 저장소, 순수 로직 유틸 | 실제 사용 |
| **unmanaged** (외부 제어) | 외부 API 호출, 메시지 브로커 발송, 이메일 발송, 시스템 시각 | mock |

**판정 기준**: 테스트 실행 중 관찰 가능한 부수효과가 **프로세스 외부**(네트워크, 파일시스템 밖, 공용 외부 시스템)로 나가는가? 나간다면 unmanaged. 프로세스 내부이거나 테스트가 완전히 제어할 수 있다면 managed.

### 공통 규칙

- **managed dependency에 호출 검증(verify) 금지** — verify는 내부 호출 순서에 결합되어, 같은 결과를 내는 리팩터링만으로 테스트가 깨진다. 최종 상태가 올바르면 과정은 중요하지 않다
- **stub(조회용 mock)에 assert 금지** — mock 검증은 나가는 명령(command)에만. 조회(query)는 반환값만 쓴다
- **포트 인터페이스를 mock하되, 구체 어댑터를 직접 사용하지 않는다** — 어댑터는 외부 의존성과의 결합 지점이므로 테스트에서 실 구현을 쓰면 느리고 불안정해진다

## 6. 구조 규칙

### 테스트 이름

비즈니스 행위를 기술한다. 메서드명·클래스명·컴포넌트명 등 구현 세부사항을 포함하지 않는다. 이름이 구현에 결합되면 리네이밍만으로 테스트가 깨진다(거짓 양성).

```
// Bad — 메서드명/컴포넌트명에 결합
"fetchAndSave - 데이터를 동기화한다"
"calculateDiscount - 할인을 계산한다"
"<OrderForm /> renders correctly"

// Good — 행위만 기술
"외부 데이터를 동기화하여 저장한다"
"VIP 등급이면 10% 할인을 적용한다"
"재고가 0이면 주문을 거부한다"
```

### Given / When / Then

세 단계로 구성한다. 주석은 선택이지만 단계 간 빈 줄로 구분한다.

```
test "재주문 시 기존 장바구니를 교체한다":
    # given
    service.placeOrder(makeOrder(productId=1))

    # when
    service.placeOrder(makeOrder(productId=2))

    # then
    orders = orderStore.listAll()
    assert orders.length == 1
    assert orders[0].productId == 2
```

### 테스트 당 하나의 행위

하나의 테스트는 하나의 비즈니스 시나리오만 검증한다. assert가 여러 개여도 검증하는 행위는 하나. 테스트가 실패했을 때 어떤 행위가 깨졌는지 즉시 알 수 있어야 디버깅이 빠르다.

### Fixture

테스트 데이터 생성용 팩토리 함수. 모든 파라미터에 기본값을 두고, 테스트에서 관련 파라미터만 명시한다.

| 상황 | 위치 |
|------|------|
| 2개 이상 테스트 파일에서 사용 | 공용 fixture 모듈 |
| 단일 테스트 파일에서만 사용 | 해당 파일 내 private 헬퍼 |

**재사용이 생기는 시점에 fixture 파일로 추출한다.** 처음부터 공용 fixture를 만들지 않는다.

## 7. 자가 평가 체크리스트

테스트 작성 후 아래 5단계를 수행한다. 각 단계에서 **해당 코드를 실제로 나열하고 판단**한다.

### 7-1. 테스트 이름 점검

작성한 모든 테스트 이름을 나열한다.

| 위반 | 조치 |
|------|------|
| 메서드명/클래스명/컴포넌트명이 포함됨 | 비즈니스 행위로 재작성 |

### 7-2. assert 점검

각 테스트의 assert를 나열한다.

| 위반 | 조치 |
|------|------|
| 프레임워크·언어·런타임 보장 동작을 검증 | 해당 assert 삭제 |
| 사소한 코드(매핑, 위임)만 검증하는 테스트 | 테스트 삭제 |

### 7-3. mock 점검

호출 검증(verify)과 mock 대상을 나열한다.

| 위반 | 조치 |
|------|------|
| managed dependency에 호출 검증 사용 | 상태 기반 검증으로 교체 |
| 구체 어댑터 클래스를 직접 사용 | 포트 인터페이스로 교체 |
| 조회(query) mock에 호출 검증 사용 | 검증 삭제, 반환값만 활용 |

### 7-4. 중복 점검

통합 테스트와 단위 테스트의 검증 범위를 비교한다.

| 위반 | 조치 |
|------|------|
| 도메인 분기를 통합 테스트에서 중복 검증 | 통합에서 삭제 |
| 같은 패턴의 다른 수준(시간별/일별) 반복 | 하나만 남기고 삭제 |
| 통합 테스트가 관리 의존성 단독 테스트가 됨 | 삭제하고 컨트롤러 통합 테스트에서 커버 |

### 7-5. 데이터 충분성 점검 (통합 테스트)

통합 테스트의 데이터 셋업과 조회 조건을 대조한다. 조회에 필터링·선택 조건이 있는데 그 조건이 실제로 후보를 걸러내는지 검증할 데이터가 없으면, 조건이 통째로 빠져도 테스트가 통과한다.

| 위반 | 조치 |
|------|------|
| 필터링 조건이 있는데 제외 대상 데이터가 없음 | 걸러져야 할 데이터 추가 |
| "최신/최대" 선택인데 후보가 1건뿐 | 이전 데이터 추가하여 선택 로직 검증 |
| 조건 경계와 무관한 데이터만 존재 | 조건 경계에 걸리는 데이터 추가 |
