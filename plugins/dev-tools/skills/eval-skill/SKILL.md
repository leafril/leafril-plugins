지정된 스킬의 SKILL.md와 참조 파일을 읽고, 아래 워크플로우로 평가한다.

## Phase 1: 스킬 읽기

`$ARGUMENTS` 처리:
- 스킬 이름이면 같은 플러그인의 `skills/{이름}/skill.md`에서 탐색한다
- 경로면 해당 경로를 직접 읽는다

1. 대상 스킬의 SKILL.md를 읽는다
2. references/ 등 참조 파일이 있으면 모두 읽는다
3. 전체 줄 수를 파악한다

## Phase 2: 구조 평가

[checklist.md](references/checklist.md)의 체크리스트를 하나씩 판정한다. 각 항목에 O/X/해당없음을 매긴다.

워크플로우 섹션(#22~25)은 스킬에 워크플로우가 없으면 모두 해당없음으로 판정한다.

## Phase 3: 내용 평가

구조와 별개로, 스킬의 **내용적 효과성**을 평가한다:

- **WHY 충분성**: 규칙/지시마다 이유가 설명되어 있는가? "MUST/NEVER" 없이 납득 가능한가? 이유 없는 규칙을 구체적으로 나열한다.
- **자유도 적절성**: 작업의 위험도/변동성에 맞는 자유도를 부여하는가? (부작용 큰 작업은 낮은 자유도, 안전한 작업은 높은 자유도)
- **워크플로우 중복**: 같은 검증을 여러 단계에서 반복하고 있지 않은가? 각 단계의 고유 역할이 명확한가?
- **경계 사례 가이드**: 가장 흔한 모호 상황에 대한 판단 기준이 있는가? 분류가 애매한 입력이 들어왔을 때 모델이 판단할 수 있는가?
- **예시 다양성**: 하나의 도메인/패턴에만 편중되지 않았는가? 다른 상황에도 규칙을 적용할 수 있게 예시가 분산되어 있는가?
- **자가 검증 실효성**: 자가 평가/검증 단계가 있다면, 구체적 행동(나열→판단)을 지시하는가? 단순 체크리스트 나열은 LLM이 기계적으로 통과시킬 수 있다.
- **호출 맥락별 분기**: 수동 호출(/명령)과 자동 호출에서 동작이 달라야 하는 부분이 있는가? (보고 상세도, 중간 확인 여부 등)

## Phase 4: 보고

아래 형식으로 보고한다:

```
## 구조 평가

| # | 항목 | 판정 | 비고 |
|---|------|------|------|
| 1 | description 구체성 | O | ... |
| 2 | ... | X | 개선 필요: ... |

통과: N/M

## 내용 평가

| 항목 | 판정 | 비고 |
|------|------|------|
| WHY 충분성 | 미흡 | §3, §5에 이유 없음 |
| ... | ... | ... |

## 개선 우선순위

1. ...
2. ...
```
