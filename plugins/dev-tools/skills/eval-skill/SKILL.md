지정된 스킬의 SKILL.md와 참조 파일을 읽고 평가한다. 두 가지 모드가 있다:

- **평가 모드** (기본): `/eval-skill <스킬>` → 구조·내용 평가 보고서 출력
- **테스트 프롬프트 모드**: `/eval-skill test-prompts <스킬>` → 개선 검증용 테스트 프롬프트만 생성

`$ARGUMENTS[0]`이 `test-prompts`이면 테스트 프롬프트 모드. 그 외에는 평가 모드.

## Phase 1: 스킬 읽기

`$ARGUMENTS` 처리 (`test-prompts` 접두사가 있으면 제거한 나머지를 스킬 경로로 사용):
- 스킬 이름이면 같은 플러그인의 `skills/{이름}/skill.md`에서 탐색한다
- 경로면 해당 경로를 직접 읽는다

1. 대상 스킬의 SKILL.md를 읽는다
2. references/ 등 참조 파일이 있으면 모두 읽는다
3. 전체 줄 수를 파악한다

## Phase 2: 구조 평가

[checklist.md](references/checklist.md)의 체크리스트를 하나씩 판정한다. 각 항목에 O/X/해당없음을 매긴다.

워크플로우 섹션(#22~25)은 스킬에 워크플로우가 없으면 모두 해당없음으로 판정한다.

## Phase 3: 내용 평가

구조와 별개로, 스킬의 **내용적 효과성**을 평가한다:

- **WHY 충분성**: 규칙/지시마다 이유가 설명되어 있는가? "MUST/NEVER" 없이 납득 가능한가? 이유 없는 규칙을 구체적으로 나열한다.
- **자유도 적절성**: 작업의 위험도/변동성에 맞는 자유도를 부여하는가? (부작용 큰 작업은 낮은 자유도, 안전한 작업은 높은 자유도)
- **워크플로우 중복**: 같은 검증을 여러 단계에서 반복하고 있지 않은가? 각 단계의 고유 역할이 명확한가?
- **경계 사례 가이드**: 가장 흔한 모호 상황에 대한 판단 기준이 있는가? 분류가 애매한 입력이 들어왔을 때 모델이 판단할 수 있는가?
- **예시 다양성**: 하나의 도메인/패턴에만 편중되지 않았는가? 다른 상황에도 규칙을 적용할 수 있게 예시가 분산되어 있는가?
- **자가 검증 실효성**: 자가 평가/검증 단계가 있다면, 구체적 행동(나열→판단)을 지시하는가? 단순 체크리스트 나열은 LLM이 기계적으로 통과시킬 수 있다.
- **호출 맥락별 분기**: 수동 호출(/명령)과 자동 호출에서 동작이 달라야 하는 부분이 있는가? (보고 상세도, 중간 확인 여부 등)

## Phase 4: 테스트 프롬프트 생성 (테스트 프롬프트 모드 전용)

**평가 모드에서는 이 Phase를 건너뛴다.** 테스트 프롬프트 모드(`/eval-skill test-prompts <스킬>`)에서만 실행한다.

Phase 1~3을 내부적으로 실행하여 개선 우선순위를 파악한 뒤, 각 개선 항목별로 검증 프롬프트를 생성한다. 구조·내용 평가 보고서는 출력하지 않고 테스트 프롬프트만 출력한다.

생성 기준:
- 개선 항목당 1개의 프롬프트. 하나의 프롬프트가 여러 개선을 동시에 검증할 수 있으면 합친다.
- 프롬프트는 실제 사용자가 입력할 법한 자연스러운 문장이어야 한다. 추상적이거나 인위적인 입력은 피한다.
- 각 프롬프트에 "무엇이 개선됐는지 확인하려면 어디를 봐야 하는지" 검증 포인트를 함께 적는다.
- 개선 사항이 없으면(모든 평가가 양호하면) "개선 사항 없음 — 테스트 프롬프트 불필요"로 안내한다.
- 테스트는 새 세션에서 실행하라고 안내한다. 현재 세션은 이미 스킬 내용을 컨텍스트에 로드한 상태이므로, 스킬 변경 사항이 반영된 결과를 정확히 확인하려면 새 세션이 필요하다.

출력 형식:
```
## 테스트 프롬프트

개선 사항이 적용된 후 **새 세션**에서 검증:

> 1. `프롬프트 내용` — 검증 포인트: ...
> 2. `프롬프트 내용` — 검증 포인트: ...
```

이 모드에서는 Phase 5 보고를 건너뛴다.

## Phase 5: 보고 (평가 모드 전용)

아래 형식으로 보고한다:

```
## 구조 평가

| # | 항목 | 판정 | 비고 |
|---|------|------|------|
| 1 | description 구체성 | O | ... |
| 2 | ... | X | 개선 필요: ... |

통과: N/M

## 내용 평가

| 항목 | 판정 | 비고 |
|------|------|------|
| WHY 충분성 | 미흡 | §3, §5에 이유 없음 |
| ... | ... | ... |

## 개선 우선순위

1. ...
2. ...
```

테스트 프롬프트가 필요하면 `/eval-skill test-prompts <스킬>`로 별도 생성한다.
