# Progress

커밋 후 자동으로 프로젝트 루트의 `progress.md`를 관리한다.
세션 간 연속성을 보장하여 다음 세션에서 맥락 없이도 작업을 이어갈 수 있게 한다.

**Why**: 에이전트 간 handoff는 파일로 남겨야 context window에 독립적이다. 진행 상황 파일이 있으면 새 세션이 git log + progress.md만 읽고 즉시 작업을 재개할 수 있다.

## 호출 맥락

이 스킬은 commit hook에서 자동 호출된다. 사용자에게 "뭘 했는지" 되묻지 않는다 — 세션 맥락과 커밋 내용에 이미 답이 있다.

## 파일 규칙

- 경로: 프로젝트 루트의 `progress.md` (단일 파일)
- 여러 기능이 동시 진행 중이면 하나의 파일 안에 기능별 섹션으로 구분
- `.gitignore`에 추가하여 커밋하지 않는다

**Why**: 단일 파일이면 새 세션에서 한 번에 전체 상황을 파악할 수 있다. 기능별로 파일을 분산하면 여러 파일을 찾아 읽어야 한다. `.gitignore`에 추가하는 이유는 임시 작업 파일이므로 이력에 남기면 노이즈가 되기 때문이다.

## 문서 구조

```markdown
# Progress

## {기능명}

### 목표
{맥락 없이 읽어도 뭘 만드는지 알 수 있는 한 줄 요약}

### 결정 사항
- {최종 결정} — {이유}

### 작업 현황
- [x] {완료된 작업}
- [ ] {다음 작업}

### 주의사항
- {다음 세션에서 알아야 할 함정, 워크어라운드, 미해결 이슈}
```

여러 기능이 진행 중이면 `## {기능명}` 섹션을 반복한다.

## 섹션별 작성 원칙

**목표**: 이 기능이 왜 필요하고 뭘 달성하려는지 한 줄로. 다음 세션의 Claude가 이것만 읽고도 방향을 잡을 수 있어야 한다.

- 좋은 예: "외부 결제 API 응답 지연 시 사용자 이탈을 줄이기 위해 비동기 결제 확인 도입"
- 나쁜 예: "결제 기능 개선" (무엇을 왜 하는지 알 수 없음)

**결정 사항**: progress의 핵심 가치. 사용자와 논의하며 합의한 설계 판단을 기록한다. 논의 과정은 생략하되, 최종 결정과 그 이유를 함께 적는다.

**Why**: 다음 세션에서 같은 질문을 다시 하지 않고, 왜 그렇게 결정했는지도 알 수 있게 하기 위함이다.

- 좋은 예: "캐시 TTL 5분 — API rate limit 고려 시 실시간 갱신은 과도"
- 나쁜 예: "Aggregate 안 쓰기로 함" (이유 없음)

**작업 현황**: 구현 단위로 체크리스트를 작성한다. 너무 세분화하지 않되, 다음 세션에서 어디서부터 이어가야 하는지 알 수 있을 정도로.

- 좋은 예: "도메인 이벤트 발행 → 리스너 구현 → 통합 테스트"
- 나쁜 예: "PaymentService.kt 수정" (코드에서 알 수 있는 정보)

**주의사항**: 다음 세션의 Claude가 밟을 수 있는 지뢰를 미리 기록한다.

- 좋은 예: "결제 취소 API는 idempotency key 필수 — 없으면 중복 취소 발생 (실제로 겪음)"
- 나쁜 예: "PaymentCancelHandler 클래스 참고" (코드 읽으면 아는 정보)

## 설계 탐색이 필요한 작업

코드 탐색이나 분석이 선행되어야 결정을 내릴 수 있는 작업에 적용하는 패턴.

**결정 사항을 비워두고 시작한다.** 탐색 전에 결정을 채우면 근거 없는 판단이 된다. 탐색 완료 후 사용자와 합의하여 채운다.

**단계별 작업 현황을 사용한다.** 작업 항목 간 의존관계가 있으면 단계로 나누고 의존관계를 명시한다.

```markdown
### 결정 사항
- (아직 없음 — 탐색 후 결정 예정)

### 작업 현황
#### 0단계: 탐색
- [ ] 현재 코드의 실제 사용 패턴 파악

#### 1단계: 설계 결정 (0단계 완료 후)
- [ ] 탐색 결과 기반으로 방향 합의

#### 2단계: 구현 (1단계 완료 후)
- [ ] 결정에 따른 코드 변경
```

**탐색 결과를 기록한다.** 탐색에서 발견한 사실, 초기 가정과 달라진 판단을 주의사항 섹션에 남긴다. 다음 세션에서 같은 탐색을 반복하지 않기 위함이다.

## 동작 흐름

커밋 후 hook으로 호출되면 다음 순서로 동작한다.

### 1단계: 상태 파악

1. `progress.md`가 있는지 확인한다
2. 있으면 읽는다
3. 현재 세션의 대화 내용과 커밋 내용을 분석한다

### 2단계: 동작 결정

| 조건 | 동작 |
|------|------|
| progress.md 없음 + 새 기능 작업 중 | → **생성** |
| progress.md 있음 + 기존 기능 섹션에 해당하는 작업 | → **갱신** |
| progress.md 있음 + 새 기능 작업 중 | → 기존 섹션과 겹치는지 판단 후 **갱신** 또는 **섹션 추가** |
| progress.md 있음 + 기능의 모든 작업 항목 완료 | → **정리 제안** |

### 3단계: 실행

**생성 시:**
1. 세션 대화에서 기능명과 목표를 파악한다
2. `progress.md`를 생성하고 문서 구조에 따라 작성한다
3. `.gitignore`에 `progress.md`가 없으면 추가한다

기능명이 불분명하면 커밋 메시지와 변경된 파일 경로에서 추론한다. 그래도 모호하면 AskUserQuestion으로 확인한다.

**갱신 시:**
1. 세션 중 합의된 결정 사항을 추가한다
2. 작업 현황의 체크리스트를 갱신한다 (완료 항목 체크, 새 항목 추가)
3. 새로 발견한 주의사항이 있으면 추가한다

세션에서 여러 기능에 걸친 작업을 했으면, 관련된 섹션을 모두 갱신한다.

**기존 섹션과 겹치는지 판단:** 커밋의 변경 파일이 기존 섹션의 작업 항목과 같은 도메인/모듈에 속하면 겹침으로 본다. 기능명은 다르지만 범위가 겹치면 기존 섹션에 통합한다.

**정리 제안 시:**
기능의 모든 작업 항목이 완료되었으면 AskUserQuestion으로 정리 여부를 확인한다. 사용자가 승인하면:
1. 결정 사항 중 영구적으로 보존할 것이 있으면 관련 docs에 반영한다
2. 해당 기능 섹션을 제거한다
3. progress.md가 비게 되면 파일 자체를 삭제한다

**Why**: 삭제는 되돌리기 어려우므로, hook에서 자동 삭제하지 않고 반드시 사용자 확인을 거친다.

### 4단계: 자가 검증

작성/갱신 후, progress.md를 다시 읽고 다음을 점검한다:
- 목표만 읽고 이 기능이 뭔지 파악할 수 있는가?
- 결정 사항에 이유 없는 항목이 있는가?
- 코드에서 알 수 있는 정보(클래스명, 변수명, 패키지 구조)를 적지 않았는가?

하나라도 해당되면 수정한 뒤 완료한다.

## 작성 규칙

- 코드에서 알 수 있는 것은 쓰지 않는다 (클래스명, 변수명, 패키지 구조)
- 설계 의도(WHY)와 도메인 맥락 중심으로 기술한다
- 프로젝트에 docs/rules/glossary.md가 있으면 해당 용어를 사용한다

**Why**: progress.md는 코드와 별개로 "왜 이렇게 했는가"를 전달하는 문서다. 코드에서 읽을 수 있는 정보를 중복 기록하면 불일치 위험이 생기고, 정작 코드에 없는 맥락(설계 의도, 도메인 배경)이 빠지게 된다.
