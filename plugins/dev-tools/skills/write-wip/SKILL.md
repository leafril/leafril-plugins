# Write WIP

`docs/wip/` 디렉토리에 진행 중인 기능의 작업 상태를 기록하는 문서를 관리한다.
세션 간 연속성을 보장하여 다음 세션에서 맥락 없이도 작업을 이어갈 수 있게 한다.

## 언제 사용하는가

| 상황 | 동작 |
|------|------|
| 새 기능 작업 시작 | WIP 문서 생성 |
| 세션 종료 / 중단 | 현재 상태로 WIP 갱신 |
| 기능 완료 | 결정 사항을 관련 docs에 반영 후 WIP 삭제 |
| 세션 시작 | `docs/wip/`에 파일이 있으면 읽고 현재 상태 파악 |

## 파일 규칙

- 경로: `docs/wip/{기능명}.md`
- 기능명: kebab-case (예: `auction-search-api.md`, `daily-stats-features.md`)
- 이름은 기능을 한눈에 알 수 있도록 짓는다

## WIP 문서 작성 가이드

### 기본 구조

```markdown
# {기능명 — 한글}

## 목표
{맥락 없이 읽어도 뭘 만드는지 알 수 있는 한 줄 요약}

## 결정 사항
- {사용자와 합의한 설계 판단}

## 작업 현황
- [x] {완료된 태스크}
- [ ] {다음 태스크}
```

### 섹션별 작성 원칙

**목표**: 이 기능이 왜 필요하고 뭘 달성하려는지 한 줄로. 다음 세션의 Claude가 이것만 읽고도 방향을 잡을 수 있어야 한다.

**결정 사항**: 이 섹션이 WIP의 핵심 가치다. 사용자와 논의하며 합의한 설계 판단을 기록한다. 논의 과정은 생략하되, 최종 결정과 그 이유를 함께 적는다. 다음 세션에서 같은 질문을 다시 하지 않고, 왜 그렇게 결정했는지도 알 수 있게 하기 위함이다.
- 좋은 예: "캐시 TTL 5분 — API rate limit 고려 시 실시간 갱신은 과도"
- 좋은 예: "Aggregate Root 지정 안 함 — 파이프라인 구조에 전통적 Aggregate 패턴 부적합"
- 나쁜 예: "캐시를 쓸지 말지 논의함"
- 나쁜 예: "Aggregate 안 쓰기로 함" (이유 없음)

**작업 현황**: 구현 단위로 체크리스트를 작성한다. 너무 세분화하지 않되, 다음 세션에서 어디서부터 이어가야 하는지 알 수 있을 정도로.

### 유연한 확장

기본 구조를 유지하되, 기능의 성격에 따라 섹션을 추가할 수 있다:
- 설계 분석 결과가 중요하면 "평가 결과" 섹션 추가
- 핵심 로직 설명이 필요하면 "핵심 설계 메모" 섹션 추가
- 우선순위별 작업이 많으면 P1/P2/P3 구조 사용
- 미래 아이디어가 있으면 "기능 아이디어 백로그" 추가

단, 항상 목표와 결정 사항은 최상단에 유지한다.

### 설계 탐색이 필요한 작업

코드 탐색이나 분석이 선행되어야 결정을 내릴 수 있는 작업(리팩토링, 아키텍처 개선 등)에 적용하는 패턴.

**결정 사항을 비워두고 시작한다.** 탐색 전에 결정을 채우면 근거 없는 판단이 된다. 탐색 완료 후 사용자와 합의하여 채운다.

**단계별 작업 현황을 사용한다.** 작업 항목 간 의존관계가 있으면 단계로 나누고 의존관계를 명시한다.

```markdown
## 결정 사항
- (아직 없음 — 탐색 후 결정 예정)

## 작업 현황

### 0단계: 탐색
- [ ] 현재 코드의 실제 사용 패턴 파악

### 1단계: 설계 결정
- [ ] 탐색 결과 기반으로 방향 합의

### 2단계: 구현
- [ ] 결정에 따른 코드 변경

## 작업 항목 의존관계
0단계 → 1단계 → 2단계
```

**탐색 결과를 기록한다.** 탐색에서 발견한 사실, 초기 가정과 달라진 판단을 섹션으로 남긴다. 다음 세션에서 같은 탐색을 반복하지 않기 위함이다.

### 문서 작성 규칙 (docs/rules/documentation.md 준수)

- 코드에서 알 수 있는 것은 쓰지 않는다 (클래스명, 변수명, 패키지 구조)
- 도메인 용어 사전(docs/rules/glossary.md)의 용어를 사용한다
- 설계 의도(WHY)와 도메인 맥락 중심으로 기술한다

## 동작별 상세

### 생성 (새 기능 시작)

1. 사용자와 기능의 목표를 확인한다
2. `docs/wip/{기능명}.md`를 생성한다
3. 목표를 작성하고, 초기 결정 사항이 있으면 기록한다
4. 코드 탐색만으로 찾기 어려운 맥락(예: 참고할 외부 문서, 특수한 설정 파일)이 있으면 기록한다

기존 WIP와 범위가 겹치면 새로 만들지 않고 기존 WIP에 추가한다.
별개의 기능이면 별도 WIP를 생성한다. 판단이 어려우면 사용자에게 확인한다.

### 갱신 (세션 종료 또는 단계 완료)

사용자가 "여기까지", "WIP 갱신" 등을 말하면, 현재 세션의 대화 내용을 분석하여 스스로 파악한다.
사용자에게 "뭘 했는지" 되묻지 않는다 — 세션 맥락에 이미 답이 있다.

세션 종료뿐 아니라, 작업의 한 단계가 완료되어 결정 사항이나 탐색 결과가 나왔을 때도 갱신한다. 세션이 길어질수록 맥락이 유실될 위험이 크므로, 의미 있는 진척이 있으면 바로 반영한다.

1. 현재 세션에서 어떤 작업을 했는지 대화 내용을 분석한다
2. 해당하는 WIP 문서를 읽는다 (없으면 새로 생성)
3. 세션 중 합의된 결정 사항을 추가한다
4. 작업 현황의 체크리스트를 갱신한다 (완료 항목 체크, 새 항목 추가)
5. 세션 중 발견한 설계 메모나 추가 맥락이 있으면 적절한 섹션에 추가한다
6. 탐색 결과로 초기 가정이 바뀌었으면 그 변경도 기록한다

세션에서 여러 WIP에 걸친 작업을 했으면, 관련된 WIP를 모두 갱신한다.

### 삭제 (기능 완료)

세션 내용을 분석하여 WIP의 작업이 완료되었는지 판단한다.
삭제 전 반드시 사용자에게 확인을 받는다 — "XX WIP를 삭제해도 될까요?"

1. 세션 내용과 WIP의 작업 현황을 비교하여 완료 여부를 판단한다
2. 사용자에게 삭제 확인을 요청한다
3. WIP에 기록된 결정 사항 중 영구적으로 보존할 것이 있는지 확인한다
4. 보존할 결정 사항은 관련 docs에 반영한다 (architecture.md, domain-design.md 등)
5. WIP 파일을 삭제한다
6. TODO에 해소된 항목이 있으면 함께 삭제한다
