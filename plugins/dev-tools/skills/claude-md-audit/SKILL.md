# CLAUDE.md Auditor

Anthropic 공식 베스트 프랙티스 기반으로 CLAUDE.md 파일을 감사하고 개선한다.

## 핵심 원칙

CLAUDE.md의 모든 줄은 이 질문을 통과해야 한다:

> **"이 줄이 없으면 Claude가 실수할까?"**

아니라면 삭제 후보다.

## 정보 레벨 분류

CLAUDE.md에 담을 정보를 세 레벨로 구분한다:

| 레벨 | 설명 | CLAUDE.md | 예시 |
|------|------|-----------|------|
| **WHAT** | 구조적 개요 | ✅ 포함 | "Redis로 세션 캐싱", "Celery + RabbitMQ로 비동기 처리" |
| **WHY** | 설계 의도 | ✅ 포함 | "REST 대신 gRPC를 쓰는 이유: 서비스 간 지연 최소화" |
| **HOW** | 구현 상세 | ❌ 제거 | 인스턴스 타입, retry 횟수, 크레덴셜 경로 |

**WHAT vs HOW 경계 판단**: 해당 정보의 존재 자체를 모르면 Claude가 탐색할 수 없다 → WHAT (포함). 존재를 알면 코드에서 상세를 찾을 수 있다 → HOW (제거).

포함/제외 판단의 상세 기준은 [references/criteria.md](references/criteria.md) 참고.

## 워크플로

### Phase 1: 탐색

프로젝트의 모든 CLAUDE.md 파일을 찾는다. 하위 디렉토리의 CLAUDE.md는 해당 디렉토리 파일 작업 시에만 로드되므로 계층 구조를 파악한다.

### Phase 2: 내용 감사

각 CLAUDE.md 파일을 읽고 모든 섹션을 다음 기준으로 평가한다:

1. **필요성** — "이 줄이 없으면 Claude가 실수할까?"
2. **정보 레벨** — WHAT/WHY/HOW 중 어디에 해당하는가?
3. **정확성** — 실제 코드베이스와 일치하는가? 다음 red flag을 명시적으로 점검:
   - **죽은 명령**: 문서의 빌드/테스트/배포 명령이 실제 `package.json`/`Makefile`/스크립트와 일치하는가
   - **죽은 경로**: 언급된 파일·디렉토리가 실제 존재하는가
   - **Stale 버전**: 명시된 언어·프레임워크 버전이 현재 설정과 일치하는가
   - **좀비 TODO**: 이미 해결된 항목이 TODO로 남아있지 않은가
   - **치환 안 된 플레이스홀더**: `<project-name>`, `<command>` 같은 템플릿 잔재 없는가

정확성 검증 시 반드시 실제 코드를 탐색하여 확인한다. 문서만 읽고 판단하지 않는다. Red flag별 점검 항목과 예시는 [references/red-flags.md](references/red-flags.md) 참고.

### Phase 3: 구조 감사

파일 단위로 구조적 품질을 검사한다:

| 기준 | 권장 |
|------|------|
| 파일 크기 | 200줄 이하 — 초과 시 준수율 하락 |
| 분할 | 긴 내용은 @import나 하위 디렉토리 CLAUDE.md로 분리 |
| 중복 | 여러 CLAUDE.md 간 동일 내용 없어야 함 |
| 충돌 | 같은 파일 내 또는 파일 간 규칙이 모순되지 않아야 함 |
| 구체성 | 모호한 지시("코드 정리해라") 없이 검증 가능한 형태여야 함 |
| 강조 | 중요 규칙에 IMPORTANT, MUST 사용 (남용 금지) |

### Phase 4: 보고서

감사 결과를 보고서로 출력한다.

```
## CLAUDE.md 감사 보고서

### 파일: ./CLAUDE.md (XX줄)

#### 제거 추천
| 줄/섹션 | 사유 |
|---------|------|
| "인스턴스 타입 t3.micro" | HOW — 코드에서 확인 가능 |

#### 수정 추천
| 줄/섹션 | 사유 |
|---------|------|
| "배포 명령: npm deploy" | stale — 현재 코드는 pnpm deploy 사용 |

#### 추가 추천
| 내용 | 사유 |
|------|------|
| Redis 캐시 존재 언급 | WHAT — 코드 탐색 중 발견, CLAUDE.md에 없음 |

#### 구조
- 줄 수: XX/200
- 충돌: 없음 / 있음 (상세)
- 모호한 지시: 없음 / 있음 (상세)
```

보고서 출력 후 사용자 확인을 받는다.

### Phase 5: 적용

사용자가 승인한 항목만 변경한다. 변경 전 diff를 보여주고 확인받는다.
