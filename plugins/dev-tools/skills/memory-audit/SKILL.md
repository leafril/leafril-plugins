# Memory Auditor

Anthropic 공식 베스트 프랙티스 기반으로 프로젝트의 CLAUDE.md 파일을 감사하고 개선한다.

## 호출 맥락

사용자가 명시적으로 요청해 호출된 경우(수동)와 다른 작업 중 자동 호출된 경우를 구분한다:

- **수동**: 보고서를 상세히 출력하고 Phase 5 편집을 정상 진행
- **자동**: Phase 4 보고서를 요약 수준으로 축약하고, Phase 5 편집은 "제거·수정 추천" 중 증거가 확실한 항목만 제안. 애매한 항목은 수동 호출로 미룬다

이유: 자동 호출은 사용자가 다른 주 작업에 집중하고 있으므로 긴 보고서·다수 편집은 주의 전환 비용이 크다.

## 핵심 원칙

CLAUDE.md의 모든 줄은 이 질문을 통과해야 한다:

> **"이 줄이 없으면 Claude가 실수할까?"**

아니라면 삭제 후보다. 이유: CLAUDE.md는 매 세션 컨텍스트를 소비하므로, 없어도 되는 줄은 순손실이다.

## 정보 레벨 분류

CLAUDE.md에 담을 정보를 세 레벨로 구분한다:

| 레벨 | 설명 | CLAUDE.md | 예시 |
|------|------|-----------|------|
| **WHAT** | 구조적 개요 | ✅ 포함 | "Redis로 세션 캐싱", "Celery + RabbitMQ로 비동기 처리" |
| **WHY** | 설계 의도 | ✅ 포함 | "REST 대신 gRPC를 쓰는 이유: 서비스 간 지연 최소화" |
| **HOW** | 구현 상세 | ❌ 제거 | 인스턴스 타입, retry 횟수, 크레덴셜 경로 |

**WHAT vs HOW 경계 판단**: 해당 정보의 존재 자체를 모르면 Claude가 탐색할 수 없다 → WHAT (포함). 존재를 알면 코드에서 상세를 찾을 수 있다 → HOW (제거).

포함/제외 판단의 상세 기준은 [references/criteria.md](references/criteria.md) 참고.

## 워크플로

### Phase 1: 탐색

프로젝트의 모든 CLAUDE.md 파일을 찾는다. 하위 디렉토리의 CLAUDE.md는 해당 디렉토리 파일 작업 시에만 로드되므로 계층 구조를 파악한다.

**구체 행동:**
- `Glob`으로 `**/CLAUDE.md`, `**/CLAUDE.local.md` 검색
- 각 파일의 줄 수 기록
- 하위 파일이 루트와 중복되는지 초기 판정용 인덱스 작성

### Phase 2: 내용 감사

각 CLAUDE.md 파일을 읽고 모든 섹션을 다음 기준으로 평가한다:

1. **필요성** — "이 줄이 없으면 Claude가 실수할까?"
2. **정보 레벨** — WHAT/WHY/HOW 중 어디에 해당하는가?
3. **정확성** — 실제 코드베이스와 일치하는가? 다음 red flag을 명시적으로 점검:
   - **죽은 명령**: 문서의 빌드/테스트/배포 명령이 실제 `package.json`/`Makefile`/스크립트와 일치하는가
   - **죽은 경로**: 언급된 파일·디렉토리가 실제 존재하는가
   - **Stale 버전**: 명시된 언어·프레임워크 버전이 현재 설정과 일치하는가
   - **좀비 TODO**: 이미 해결된 항목이 TODO로 남아있지 않은가
   - **치환 안 된 플레이스홀더**: `<project-name>`, `<command>` 같은 템플릿 잔재 없는가

**정확성 검증은 반드시 실제 코드 탐색으로 수행한다** (이유: 문서끼리만 비교하면 양쪽이 함께 stale인 경우를 놓친다). 각 주장에 대해 다음 최소 증거를 수집한다:

| 문서 주장 | 검증 행동 |
|----------|----------|
| 빌드/테스트/배포 명령 | `package.json`의 `scripts`, `Makefile`, `justfile` 등을 `Read`로 열어 해당 명령 존재·일치 확인 |
| 언급된 경로·디렉토리 | `Glob`으로 실제 존재 확인 |
| 언급된 프레임워크·언어 버전 | `.nvmrc`, `engines`, `pyproject.toml`, `go.mod` 등에서 `Read`로 대조 |
| 언급된 아키텍처 구성요소 (Redis, Kafka 등) | 해당 의존성/설정이 `package.json`, `requirements.txt`, `docker-compose.yml` 등에 있는지 `Grep` |
| TODO·"임시" 표현 | 관련 코드를 `Grep`해서 실제로 미해결인지 확인 |

"확인했다"고 보고할 때는 위 표에서 어떤 증거를 수집했는지 보고서에 함께 기록한다. Red flag별 상세 예시는 [references/red-flags.md](references/red-flags.md) 참고.

### Phase 3: 구조 감사

파일 단위로 구조적 품질을 검사한다:

| 기준 | 권장 | 이유 |
|------|------|------|
| 파일 크기 | 200줄 이하 | 초과 시 지시 준수율 하락 |
| 분할 | 긴 내용은 @import나 하위 디렉토리 CLAUDE.md로 분리 | 관련 작업 시에만 로드되어 컨텍스트 절약 |
| 중복 | 여러 CLAUDE.md 간 동일 내용 없어야 함 | 수정 시 한쪽만 바뀌어 문서간 drift 발생 |
| 충돌 | 같은 파일 내 또는 파일 간 규칙이 모순되지 않아야 함 | 모순 시 Claude 행동 예측 불가 |
| 구체성 | 모호한 지시("코드 정리해라") 없이 검증 가능한 형태 | 해석을 추측해야 하므로 준수율 하락 |
| 강조 | 중요 규칙에만 IMPORTANT, MUST 사용 | 남용 시 강조 신호가 마비됨 |

### Phase 4: 보고서 및 승인

감사 결과를 보고서로 출력한다.

```
## CLAUDE.md 감사 보고서

### 파일: ./CLAUDE.md (XX줄)

#### 제거 추천
| 줄/섹션 | 사유 | 증거 |
|---------|------|------|
| "인스턴스 타입 t3.micro" | HOW — 코드에서 확인 가능 | `terraform/main.tf`에 동일 값 존재 |

#### 수정 추천
| 줄/섹션 | 사유 | 증거 |
|---------|------|------|
| "배포 명령: npm deploy" | stale | `package.json` scripts에 `deploy: pnpm ...` |

#### 추가 추천
| 내용 | 사유 | 증거 |
|------|------|------|
| Redis 캐시 존재 언급 | WHAT — CLAUDE.md에 없음 | `docker-compose.yml`에 redis 서비스 |

#### 구조
- 줄 수: XX/200
- 충돌: 없음 / 있음 (상세)
- 모호한 지시: 없음 / 있음 (상세)
```

보고서 출력 후 **`AskUserQuestion`으로 각 카테고리(제거/수정/추가)별로 적용 범위를 확인받는다.** 텍스트 번호 목록으로 "1,3번 적용"을 받는 방식은 쓰지 않는다 (파싱 실수 위험, 선택지가 UI로 노출 안 됨). 카테고리별로 "전부 적용 / 일부 선택 / 건너뛰기" 옵션을 제시하고, "일부 선택"일 때만 개별 항목을 `AskUserQuestion`으로 다시 묻는다.

### Phase 5: 적용

승인된 항목만 `Edit`으로 변경한다. 변경 전 각 편집의 diff를 보여주고, 의미를 바꾸는 편집(단순 오타·포맷 제외)은 `AskUserQuestion`으로 "적용 / 수정 / 취소"를 한 번 더 확인받는다.

### Phase 6: 재감사 루프

편집 후 반드시 재감사한다. 편집 행위 자체가 새로운 문제를 만들 수 있기 때문이다 (예: 줄 삭제로 문맥이 끊김, 수정한 명령이 또 다른 stale 참조를 유발).

재감사 항목 (이것만 재확인, 전체 감사 반복 금지 — 중복 방지):
- 편집된 파일의 줄 수가 200 이하인가
- Phase 3의 중복·충돌·모호 지시가 해소되었는가
- 수정·추가한 줄이 Phase 2 red flag에 다시 걸리지 않는가

**탈출 조건**: 위 3개 모두 통과.
**실패 시**: 실패 항목만 가지고 Phase 5로 복귀해 재편집. 같은 파일에서 2회 연속 실패하면 루프를 멈추고 사용자에게 상황을 보고한 뒤 지시를 받는다 (무한 루프 방지).
