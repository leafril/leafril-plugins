# 캐시 관리

config 파일 내부 `cache` 섹션에 스키마·테이블 메타데이터를 저장한다. 매 쿼리마다 `information_schema` 조회를 생략하여 응답 속도를 높이고, 테이블 설명으로 자연어 매칭 정확도를 개선한다.

## 캐시 구조

```json
{
  "cache": {
    "psql-dev": {
      "schemas": ["public", "domain"],
      "tables": {
        "public": {
          "users": "사용자",
          "accounts": "미사용 (레거시)",
          "orders": "주문",
          "scan_result": ""
        }
      }
    }
  }
}
```

- 캐시 키: DB 별칭 단위
- `schemas`: 스키마 이름 배열
- `tables`: 스키마별 `{ 테이블명: 설명 }` 객체. 설명이 없으면 빈 문자열 `""`
- 테이블 설명은 자연어 → 테이블 매칭 시 사용된다. 설명이 있으면 테이블 이름보다 설명을 우선 참조한다

## 스키마 조회 (캐시 미스 시)

캐시에 해당 별칭의 `schemas`가 없으면 메타쿼리로 조회 → 캐시에 저장:

```sql
SELECT schema_name FROM information_schema.schemata
WHERE schema_name NOT IN (EXCLUDE_LIST)
ORDER BY schema_name;
```

| 엔진 | EXCLUDE_LIST |
|------|-------------|
| psql | `'pg_catalog', 'information_schema', 'pg_toast'` |
| mysql | `'information_schema', 'performance_schema', 'mysql', 'sys'` |

## 테이블 조회 (캐시 미스 시)

캐시에 해당 스키마의 `tables`가 없으면 메타쿼리로 조회 → 캐시에 저장 (설명은 빈 문자열):

```sql
SELECT table_name FROM information_schema.tables WHERE table_schema = 'SCHEMA' ORDER BY table_name;
```

## 테이블 매칭 규칙

1. 테이블이 0개면 → 스키마가 비어있다고 안내 → 다른 스키마 선택 제안
2. **캐시에 설명이 있는 테이블은 설명을 우선 참조하여 매칭한다.** 예: 사용자가 "사용자"라고 하면 `users: "사용자"` 설명이 있는 테이블을 선택하고, 설명이 "미사용"인 `accounts`는 제외한다
3. 설명이 없는 테이블은 이름으로 매칭한다
4. 컬럼 정보가 필요하면 추가 조회 (psql: `\d 테이블명`, mysql: `SHOW COLUMNS FROM 테이블명`)
5. 테이블을 찾지 못하면 → 사용자에게 테이블명 질문

### 모호한 매칭 해소

자연어의 한 개념에 테이블 후보가 **2개 이상** 매칭되고, 캐시 설명으로 구분할 수 없으면 `AskUserQuestion`으로 질문한다:

```
AskUserQuestion:
  question: "'사용자'에 해당하는 테이블이 어느 쪽인가요?"
  options:
    - label: "users"
      description: "email: john@example.com 등"
    - label: "accounts"
      description: "login_id: admin01 등"
```

- 각 후보 테이블에서 샘플 데이터를 1~2건 조회하여 description에 표시한다
- 사용자가 선택하면 **캐시의 테이블 설명을 자동 업데이트**한다
- 다음 쿼리부터는 캐시 설명으로 즉시 매칭되므로 질문이 반복되지 않는다

## 캐시 생성 (자동)

최초로 스키마 또는 테이블 목록을 DB에서 조회할 때 config 파일의 `cache` 섹션에 자동 저장한다.

1. 스키마 조회 결과 → `cache[별칭].schemas` 배열로 저장
2. 테이블 조회 결과 → `cache[별칭].tables[스키마]` 객체로 저장 (설명은 빈 문자열 `""`)
3. config 파일을 Write로 업데이트

다른 별칭의 캐시 정보를 참고하여 쿼리를 생성하더라도, 현재 별칭에 캐시가 없으면 쿼리 실행 후 스키마·테이블 목록을 캐시에 저장한다.

## 캐시 갱신 (`/sql refresh`)

사용자가 `/sql refresh` 또는 `/sql refresh <별칭>`을 실행하면:

1. 대상 DB에 메타쿼리로 스키마·테이블 목록을 새로 조회
2. **기존 테이블 설명은 보존한다.** 새로 추가된 테이블만 빈 문자열로 추가하고, 삭제된 테이블은 제거한다
3. config 파일을 Write로 업데이트
4. 갱신 결과 요약을 출력한다 (스키마 수, 테이블 수)

## 설명 자동 업데이트

다음 상황에서 테이블 설명을 자동 업데이트한다:

| 트리거 | 동작 |
|--------|------|
| 모호한 매칭에서 사용자가 테이블 선택 | 선택된 테이블에 자연어 개념을 설명으로 저장. 선택되지 않은 후보에는 부정 설명 저장 (예: "기관 아님") |
| 사용자가 대화 중 테이블 역할을 언급 | 해당 테이블의 설명을 업데이트 |

| 사용자 발화 | 캐시 업데이트 |
|-------------|---------------|
| "accounts는 레거시야" | `"accounts": "미사용 (레거시)"` |
| "users가 사용자 테이블이야" | `"users": "사용자"` |
| "orders는 주문 내역이야" | `"orders": "주문 내역"` |

설명 업데이트 시 config 파일의 `cache` 섹션을 즉시 Write한다.

## 경계 사례

| 상황 | 판단 기준 |
|------|-----------|
| 캐시 설명과 사용자 표현이 부분 매칭 (예: 설명="사용자", 요청="회원") | 같은 도메인 개념을 가리키는 동의어·유의어면 매칭 후보로 포함 (사용자↔회원, 주문↔결제). 단순 글자 유사도가 아닌 도메인 의미 기준. 후보가 2개 이상이면 AskUserQuestion |
| 부정 설명 테이블을 사용자가 명시적으로 지정 (예: "accounts 테이블 조회해줘") | 부정 설명을 무시하고 그대로 실행. 사용자가 테이블명을 직접 지정하면 캐시 설명보다 우선 |
| 캐시 설명이 있지만 사용자 요청이 어떤 테이블에도 매칭되지 않음 | 사용자에게 테이블명을 질문. 캐시가 있어도 강제 매칭하지 않음 |
| 사용자 피드백이 기존 설명과 충돌 (예: 기존="주문", 피드백="결제 내역") | 최신 피드백으로 덮어씀. 사용자의 가장 최근 발화가 정답 |
